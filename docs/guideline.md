# IOC 需求设计指南

> 从设计稿/需求文档到可编码的前端方案，分三步渐进收敛。
> 每一步只回答一个核心问题，下游不重复上游已收敛的结论。

---

## 思考顺序

```
输入（设计稿 / 需求文档 / 存量指标 / 项目代码）
  │
  ▼  Step 1 — 需求规格
  │  回答：做什么、不做什么、怎样算做完
  │  产出：Spec.md + Acceptance.md
  │
  ▼  Step 2 — 数据方案
  │  回答：指标口径是什么、数据从哪来、展示位拿什么形状的数据
  │  产出：Data-Dev.md + Display-Contract.md + Mock-Spec.md
  │
  ▼  Step 3 — 前端方案
  │  回答：组件怎么拆、复用什么、任务怎么切
  │  产出：Frontend-Plan.md + Testability-Map.md
  │
  ▼  拆 task / 编码 / 自动化测试
```

---

## 每步做什么、不做什么

| 步骤 | 视角 | 核心产出 | 禁区 |
| --- | --- | --- | --- |
| 需求规格 | 产品/用户 | 功能需求列表 + GWT 验收场景 | 不写技术术语（无 API、数据库、框架） |
| 数据方案 | 数据/对接 | 指标清单 + 展示位数据契约 + Mock 场景 | 不写 SQL、不定义 API、不定义刷新策略 |
| 前端方案 | 前端工程 | 组件树 + 结构决策 + 任务切片 + 测试映射 | 不写实现代码 |

---

## 边界切分

最容易混淆的三组边界：

| 问题 | 归属 |
| --- | --- |
| 用户能看到什么、操作什么 | 需求规格 |
| 这个数值怎么算、数据从哪张表来 | 数据方案 |
| 这个组件是新建还是复用、怎么刷新、空态怎么展示 | 前端方案 |

展示位相关的细分：

| 关注点 | 归属 |
| --- | --- |
| 展示位对应哪个指标、什么数据形状、新鲜度等级 | 数据方案 |
| 刷新机制（轮询/WebSocket/手动）、空态/错误态展示 | 前端方案 |

---

## 每步的核心工作模式

三步共享相同的工作节奏：

### 1. 盘点（交叉比对 + 分类）

把输入源交叉对照，产出分类清单：
- 需求规格：需求文档 × 设计稿 → 需求地图（已对齐 / 缺 UI / 缺文档 / 冲突）
- 数据方案：数据需求 × 存量指标 → 指标清单（复用 / 修改 / 新增）
- 前端方案：组件需求 × 存量组件 → 组件清单（复用 / 扩展 / 新增）

### 2. 澄清（补信息 + 消歧义）

按维度扫描信息完整度，只问真正缺失的：

| 步骤 | 澄清维度 |
| --- | --- |
| 需求规格 | 功能边界 · 角色权限 · 业务规则 · 交互流程 · 数据生命周期 · 异常边界 · 术语一致性 |
| 数据方案 | 数据口径 · 源表定位 · 主题表映射 · 时间粒度 · 维度切面 · 计算逻辑 |
| 前端方案 | 状态管理 · API 对接 · 扩展兼容性 · 图表库选型 · 测试约束 |

问题标记规则：
- 🔴 缺失关键信息，必须回答，否则阻断生成
- 🟡 有证据可推测，给出推荐选项，可确认/跳过（标为工作假设）
- 🟢 已确认，不再提问

### 3. 生成（产出文档）

基于盘点 + 澄清的结论，生成结构化文档。每份文档的每条结论必须可溯源到输入材料。

## 关键规则

### 不编造

- 设计稿中没有的 UI 元素，不在文档中描述
- 源表信息不确定时标记 `[待确认]`，不猜测字段名
- 组件复用必须有代码证据，不凭名称相似直接判定

### 不确定性显式传播

`[待确认]` 标记从上游一路传播到下游：
- 需求规格中标 `[待确认]` 的 FR → 对应的验收场景也标 `[待确认]`
- 数据方案消费这些 FR → 关联指标也标 `[待确认]`
- 前端方案中未审批的扩展/新增组件 → 标 `[待确认:需开发者审批]`

不确定性不被静默吞掉。

### 降级而非阻断

输入不完整时按精度降级，而非全部阻断：

| 缺失输入 | 降级方式 |
| --- | --- |
| 无设计稿 | 跳过 UI 交叉引用，仅基于需求文档 |
| 无存量指标 | 所有指标按"新增"处理 |
| 无数据契约 | 前端方案可出，但不做指标级闭环 |
| 无需求规格 | **阻断**（这是唯一的硬阻断） |

### 用户可提前终止

任何步骤的澄清阶段，用户可以说"够了"：
- 未回答的 🔴 标记为 `unresolved`
- 未确认的 🟡 标记为 `assumed`
- 均写入最终文档，不当作已确认的需求

## 产出文件清单

```
docs/<需求名>/
├── Spec.md                 需求规格（做什么、不做什么）
├── Acceptance.md           GWT 验收场景
├── Data-Dev.md             指标口径与数据链路
├── Display-Contract.md     展示位-指标映射与数据形状
├── Mock-Spec.md            最小覆盖的 Mock 数据方案
├── Frontend-Plan.md        组件清单、结构决策、任务切片
└── Testability-Map.md      测试定位方案（Slot-ID + 选择器）
```

---

## 一句话

> 需求规格收敛"做什么"，数据方案收敛"数据怎么流"，前端方案收敛"代码怎么拆"。
> 上游不碰下游领域，下游不重复上游澄清。每份文档都是一个确定性边界。
