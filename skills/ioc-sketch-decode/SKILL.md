# 设计稿视觉解码

## 概述

这个 skill 把设计稿截图转化为结构化的文字线框图。它专为推理能力有限的多模态模型设计——只要求"描述所见"，不要求"分析所需"。

核心原则：**描述，不解读。** 每一条输出都是对画面的客观记录，不是对需求的推断。

产出的 `Sketch-Decode.md` 是 `ioc-spec` 的前置输入。它必须足够完整，使纯文本模型仅凭这份文档就能理解设计稿中的所有可见信息。

## 硬门禁

1. **不要推断功能目的。** 看到搜索框就写 `input "搜索工单编号或标题"`，不要写"用于搜索订单的输入框"。
2. **不要遗漏任何可见文字。** 所有文案直接嵌入线框图，不另行转录。
3. **不要编造不可见的内容。** 下拉框收起时，只记录可见的默认值，不要猜测完整选项列表。
4. **不要合并截图。** 每张截图独立一个区块，即使内容相似。
5. **不要输出设计建议。** 这不是设计评审，是视觉转录。

## 何时使用

适用场景：

- 收到设计稿截图，需要为后续需求分析提供结构化的视觉描述
- 设计平台/低代码平台可导出页面结构 JSON，需要转换为标准化布局描述
- 截图 + 导出 JSON 同时提供，需要交叉校验

不适用场景：

- 没有设计稿也没有导出 DSL，只有文字需求（直接用 `ioc-spec`，跳过本 skill）
- 需要对设计稿做交互评审或视觉走查（不是本 skill 的职责）

## 输入

| 输入 | 必须 | 说明 |
| --- | --- | --- |
| 设计稿截图 | 至少一种 | 1~N 张，用户在对话中提供 |
| 导出 DSL JSON | 至少一种 | 设计平台/低代码平台导出的页面结构 JSON（格式见 [references/dsl-mapping.md](references/dsl-mapping.md)） |
| 需求名称 | ✓ | 用于 `docs/<需求名>/Sketch-Decode.md` 路径 |
| 项目组件文档 | 可选 | 前端仓库提供的功能化组件描述 |

> **截图 vs 导出 DSL**：两者至少提供一种。都提供时，导出 DSL 作为布局还原的主数据源，截图用于补充可视化细节（颜色、状态、模糊区域）。

## 工作流

**没有多轮交互。** 用户提供输入后，一次性产出完整的 Sketch-Decode.md。

### 步骤 1：确认输入

- 确认用户提供的输入类型：截图、导出 DSL JSON、或两者都有
- 确认需求名称（用于 `docs/<需求名>/Sketch-Decode.md` 路径）
- 如果有截图，为每张截图获取用户给的标题（如"列表页"、"新建弹窗"、"详情页"）
- 确认是否有项目组件文档可供匹配

### 步骤 2：逐页面解析

**有导出 DSL JSON 时**：将 JSON 结构转换为布局 DSL（映射规则见 [references/dsl-mapping.md](references/dsl-mapping.md)），结合截图（如有）补充颜色、状态等可视化信息，然后完成剩余 3 节。

**仅有截图时**：对每张截图，严格按照 [4 节 Schema](references/decode-schema.md) 逐节填写。

### 步骤 3：写入文件

产出写入 `docs/<需求名>/Sketch-Decode.md`。

文件结构：

```markdown
# Sketch-Decode: <需求名>

> 由 ioc-sketch-decode 生成 | <日期>

---

## 截图 1: <用户给的标题>

### 布局还原
（布局 DSL，包含所有可见文字和数据）

### 组件识别
（功能描述 + 项目组件匹配）

### 交互与状态
（可交互元素 + 状态标识）

### 解析自检
（模糊区域和不确定项）

---

## 截图 2: <用户给的标题>

（同上 4 节结构）
```

### 步骤 4：完成

```
✓ ioc-sketch-decode 完成
输入来源：截图 / 导出 DSL / 两者
产出：docs/<需求名>/Sketch-Decode.md
下一步：在文本 agent 中运行 ioc-spec，基于 Sketch-Decode.md + 需求文档产出 Spec.md 和 Acceptance.md
```

## 4 节 Schema 速查

完整定义和示例见 [references/decode-schema.md](references/decode-schema.md)。

| 节 | 做什么 | 不做什么 |
| --- | --- | --- |
| 布局还原 | 用布局 DSL 还原页面结构，嵌入所有可见文字和数据 | 不评价布局合理性，不遗漏文案 |
| 组件识别 | 描述功能特征，匹配项目组件（如有组件文档） | 不使用前端技术术语 |
| 交互与状态 | 列出可交互元素和可见状态标识 | 不描述点击后的预期行为 |
| 解析自检 | 标注模糊区域和不确定项 | 不忽略或静默跳过 |

## 多截图处理

- 每张截图独立一个 `## 截图 N: <标题>` 区块
- 截图之间不做关联推断（如"截图 2 的弹窗是从截图 1 的按钮打开的"——这是 ioc-spec 的工作）
- 如果同一页面有多个状态截图（如空态、加载态、错误态），每个状态独立一个截图区块

## 项目组件文档

如果前端仓库提供了功能化组件描述文档，本 skill 会在"组件识别"节中将截图中的区域与已有组件做匹配。

组件文档要求：
- 用功能语言描述组件（"可筛选的数据展示区"），不使用技术术语（不写 props、API、框架名）
- 文档路径由用户在调用时告知，或通过 MCP 读取
- 详细格式规范见 [references/decode-schema.md](references/decode-schema.md) 的"组件文档的要求"部分

## 渲染脚本

布局 DSL 可通过渲染脚本转换为 ASCII wireframe 用于人工校对：

```bash
node skills/ioc-sketch-decode/scripts/render-wireframe.mjs docs/<需求名>/Sketch-Decode.md
```

脚本从 Sketch-Decode.md 中提取所有"布局还原"代码块，渲染为可视化线框输出到 stdout。不修改原文件。无额外依赖。
