# 知识反哺协议

定义 ioc-data-plan 运行结束后如何将新发现的经验写回 [knowledge-base.md](knowledge-base.md)。

---

## 与最终输出的关系

知识反哺是**独立于最终三行索引的后续动作**，不挂在最终输出上。

执行顺序：
1. 输出三行索引（P5），skill 正式完成
2. 在下一个交互轮次中，检查是否有可沉淀的经验
3. 如有，输出知识更新建议，等待用户决策
4. 如无，不产生额外输出

## 触发时机

在三行索引输出后的下一轮交互中，回顾本次 skill 运行全过程。

## 检查清单

按以下三个维度提取候选知识：

| 维度 | 检查什么 | 写入知识库的哪个区 |
|------|---------|-------------------|
| ETL 模式 | Phase 2 澄清中是否出现了知识库中没有的转换模式 | 一、ETL 模式库 |
| 领域口径 | Phase 2 澄清中是否出现了特殊的口径定义或行业特有的指标体系 | 二、领域指标模板 |
| 源表资产 | 本次是否获取了新的源表结构信息 | 三、源表资产 |

## 输出格式

```
---
**[知识更新建议]** 本次运行发现以下可沉淀的经验：

| # | 类型 | 内容摘要 | 建议写入 |
|---|------|---------|---------|
| K-1 | ETL 模式 | "近7日"不含今天是本项目的通用约定 | P-002 注意事项补充 |
| K-2 | 领域口径 | SLA 达标标准为响应<4h 且解决<24h | C-001 口径注意补充 |
| K-3 | 源表资产 | ods_workorder 表结构已获取 | 三区新增 ST-001 |

→ 确认写入 / 跳过 / 修改后写入
---
```

## 写入规则

用户确认后，按以下规则写入 knowledge-base.md：

1. **追加不覆盖**：新条目追加到对应区的末尾，不修改已有条目
2. **编号连续**：ETL 模式用 `P-NNN`，领域模板用 `C-NNN`，源表资产用 `ST-NNN`，编号递增
3. **标注来源**：每条新知识附带元数据

```markdown
> 来源：<需求名> | <日期> | ioc-data-plan 反哺
```

4. **修改已有条目**：如果是对已有条目的补充，在该条目的"注意事项"列表末尾追加，同样标注来源
5. **冲突处理**：如果新知识与已有知识矛盾，不自动覆盖，在新条目中标注 `[与 P-00N 存在差异]`，由用户决定保留哪个

## 不触发的情况

以下情况不产生知识更新建议（下一轮交互中直接跳过）：

- 本次运行完全在知识库已有范围内，无新发现
- Phase 2 未发生（所有指标均为复用）
- 用户提前终止，澄清信息不足以形成可靠经验

## 知识质量标准

候选知识必须满足以下条件才能建议写入：

- **可复用**：不是只适用于本次需求的特例，而是可能在未来需求中复现的模式
- **已验证**：经过用户确认（🟢），而非仅是工作假设（🟡）
- **可操作**：描述具体到可直接指导下次决策的程度，不是泛泛的原则

## 知识隔离

知识库中的经验带有来源标注（需求名 + 日期）。使用时注意：

- ETL 模式（P 区）：通常可跨项目复用
- 领域口径（C 区）：可能带项目/行业特异性，使用时应与当前项目的上下文交叉验证，不直接作为默认值
- 源表资产（ST 区）：强绑定特定项目的数据环境，跨项目复用需确认数据源一致性
