# Display-Contract.md 模板

以下是 `Display-Contract.md` 的标准结构。所有章节必须保留，无内容的章节标注"不适用"并说明原因。

本文档的字段列 **只能引用 Data-Dev.md 里已通过 Phase 3 确认门的表/字段**，严禁在此处首次引入新字段名。如果 Phase 3 采用了无表设计降级，本文档自动进入 **Lite 模式**——只定义逻辑契约（展示位 → 指标 → 值形状 → 新鲜度），不含字段列。

## Lite 模式（Phase 3 表设计未确认时使用）

触发条件：Phase 3 用户选择"由数据工程师后续定表设计"，或 Data-Dev.md 中表结构被标记为 `[待确认:需数据工程师定稿]`。

**与完整模式的区别**：

- **省略 `应用表.字段` 列**
- **只保留逻辑层**：展示位 → 指标语义名 → 值形状 → 新鲜度
- 在文档头部显式标注 `> 模式: Lite（物理字段绑定待数据工程师定稿后补齐）`

# 展示数据契约: <需求名>

## 写作规范

1. **展示位命名一致**：展示位列的命名必须与 Sketch-Decode.md 中布局 DSL 的元素名逐字一致。不允许自行改名。
2. **字段必须已确认**（完整模式）：`应用表.字段` 列必须在 Data-Dev.md 中出现过，不允许在本文档首次引入新字段名。这是防止"编造字段"的硬约束。
3. **降级到 Lite 而非阻断**：Phase 3 无表设计时进入 Lite 模式，保留交付能力。不要在没有真实表设计时强行填字段列。
4. **计算显式化**：任何不是直接读取应用表字段的数据，都必须在"计算指标"章节写明公式。不允许隐含计算。
5. **值形状明确**：值类型列标注数据的编程类型（integer / decimal / percentage / string / integer[] 等），格式/单位列标注前端格式化所需信息。
6. **不涉及前端行为**：本文档只定义"展示位需要什么数据、什么形状、什么新鲜度"。刷新机制、空态展示、异常处理、加载态由 ioc-frontend-plan 承担。
7. **不涉及 API 设计**：本文档不定义 API 路径、请求参数、响应结构等技术接口。

## 与下游的衔接

无论完整模式还是 Lite 模式，`ioc-frontend-plan` 都能直接消费：

| 前端方案关心的 | 来自本文档的 |
| --- | --- |
| 这个展示位要展示什么指标 | 数据指标列 |
| 数据的值形状与格式 | 值类型 + 格式/单位 |
| 多久更新一次（决定刷新策略） | 数据新鲜度等级 |
| 衍生值怎么算 | 计算指标 |

Lite 模式下缺的只是"字段绑定"，前端方案的组件结构、Task 切分、Testability-Map 都不依赖它。
