<!-- 每张新增/修改的应用表一个子章节，必须含 DDL 草稿。 -->

### T-1: dws_workorder_daily（新建）

- **分层**：DWS
- **业务域**：工单
- **时间粒度**：日级（按 `dt` 分区）
- **维度切面**：按 region_id 拆分
- **数据来源**：`ods_workorder` 清洗聚合
- **调度频率**：每日凌晨 01:00
- **承载指标**：DI-1（当日工单总数）、DI-2（当日待处理数）
- **预计行数**：region_id 数 × 天数（日增量 ~ 30 行）

**DDL 草稿**：

```sql
CREATE TABLE `dws_workorder.dws_workorder_daily` (
  `region_id` string COMMENT '区域编码',
  `total_count` bigint COMMENT '工单总数',
  `pending_count` bigint COMMENT '待处理工单数',
  `completed_count` bigint COMMENT '已完成工单数'
) COMMENT '工单日粒度聚合表'
PARTITIONED BY (`dt` string COMMENT '日期 yyyy-MM-dd')
ROW FORMAT SERDE 'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS ORC;
```

**开发要点**：

- 聚合 key：`dt + region_id`
- 行裁剪：`WHERE dt = '${yyyy-MM-dd}'`，避免全表扫描
- NULL 处理：region_id 可能为 NULL，按规范转为空字符串
- 分区新增字段时需加 `CASCADE`

### DM-1: SLA 达标率

- **现有口径**：月度达标率 = 月内达标工单 / 月内总工单
- **目标口径**：日度达标率 = 当日达标工单 / 当日总工单（达标标准：响应时间 < 4h）

- **归属表.字段**：`app_workorder.sla_rate`（见 T-2 修改）
- **影响范围**：月度报表 `monthly_ops_report` 需要同步调整引用
- **开发要点**：
  - 新增字段需要历史回刷（由调度链路追加 backfill 任务）
  - 老字段保留 2 个月后下线

## 写作规范

1. **口径精度**：口径定义必须包含：筛选条件（WHERE）、聚合函数（COUNT/SUM/AVG）、时间范围、分组维度。不允许出现"统计相关数据"等模糊表述。
2. **不写 SQL 业务逻辑，但要写 DDL 草稿**：本文档只定义"要算什么"以及"承载指标的表长什么样"，不写业务 SQL。SQL 实现留给下游 skill 或人工开发。DDL 草稿是表结构规格的一部分，属于必写内容。
3. **时间粒度 / 维度切面不再作为指标独立属性**：它们已在"表结构设计"里由 DDL 承载（时间粒度 = 分区字段 / 表名后缀；维度切面 = 宽表维度列）。在指标子章节中不再重复列出，只在表子章节中说明一次。
4. **可溯源**：每个指标标注关联的页面组件（来自 Sketch-Decode.md）和功能需求（来自 Spec.md）；每张表标注承载的指标编号。
5. **标记不确定**：源表/表设计信息不足时，用 `[待确认:原因]` 或 `[推测:需数据工程师确认]` 明确标记，不编造不猜测。DDL 草稿信息不全时，保留骨架但用注释或占位符标注。
6. **影响分析**：修改类指标必须评估影响范围，防止口径变更导致其他页面/报表出错。
7. **DDL 规范沿用企业数据开发规范**：表名带分层前缀（ods/dwd/dws/ads）、ORC 存储、分区字段显式、字段含 COMMENT。具体可参考 `docs/数据开发流程.md`。
