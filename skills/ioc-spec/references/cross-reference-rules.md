# 交叉引用与溯源规则

ioc-spec 的核心质量保障机制：每条需求和每个验收场景都必须可溯源到输入材料。

---

## 溯源链

```
Sketch-Decode.md ──┐
                    ├──→ Spec.md (FR-N, 来源列) ──→ Acceptance.md (AC-N ↔ FR-N)
需求文档 ──────────┘
```

## 规则

### R1: 功能需求必须标注来源

Spec.md 中每条 FR 的"来源"列必须指向以下至少一个：

- `Sketch-Decode 截图N: <组件/元素名>`
- `需求文档 X.X 节`
- `用户澄清 Q-N`（Phase 2 中用户回答的问题编号）

如果无法指向任何来源，标记 `[待确认:无视觉/文档依据]`。

### R2: 验收场景与功能需求一一对应

Acceptance.md 中每个 AC-N 必须对应 Spec.md 中的 FR-N。

- 不允许存在没有 FR 对应的 AC
- 不允许存在没有 AC 对应的 FR（除非 FR 标记了 `[待确认]`，此时 AC 也标记 `[待确认]`）

### R3: 界面元素引用必须一致

Acceptance.md 的 GWT 场景中引用的界面元素（按钮文案、页面名称、字段标签等），必须与 Sketch-Decode.md 中"逐字文案"节的原文一致。

- 正确：`用户点击"新建工单"`（与 Sketch-Decode.md 一致）
- 错误：`用户点击"创建工单"`（Sketch-Decode.md 中写的是"新建工单"）

### R4: 禁止编造 UI 细节

以下情况属于编造：

- 描述 Sketch-Decode.md 中未出现的页面或组件
- 为 Sketch-Decode.md "解析自检"中标记为不可见的元素赋予具体内容（如收起的下拉框选项）
- 推断截图之间的导航关系（如"从列表页点击跳转到详情页"——除非 Sketch-Decode.md 中有明确的导航线索）

正确做法：

- 需求文档提到但截图中未体现的功能 → FR 中只记录文字需求，来源标注为需求文档，不描述 UI 表现
- Sketch-Decode.md 中标记不确定的内容 → FR 中使用 `[待确认:Sketch-Decode 解析自检标注]`

### R5: 待确认项传播

如果 FR 标记了 `[待确认]`，对应的 AC 场景也必须标记 `[待确认]`，不能为一个不确定的需求写出确定的验收标准。

## 校验清单

Phase 4 完成后，按以下清单自检：

- [ ] 每条 FR 有来源标注
- [ ] 每条 FR 有对应的 AC
- [ ] AC 中引用的界面元素与 Sketch-Decode.md 逐字文案一致
- [ ] 无编造的 UI 细节
- [ ] 待确认项在 FR 和 AC 中同步标记
- [ ] Spec.md 和 Acceptance.md 中无技术术语
