## 写作规范

1. **语言**：所有功能描述使用"用户可以..."、"系统显示..."、"页面展示..."的句式。禁止出现 API、数据库、接口、字段类型、框架名等技术术语。
2. **粒度**：一个 FR 对应一个可独立验收的功能点。不要把多个操作塞进一个 FR，也不要把一个操作拆成过细的多个 FR。
3. **来源**：来源列是本文档与 Sketch-Decode.md / 需求文档之间的追溯链。缺少来源的 FR 必须标记 `[待确认]`。
4. **不做的事**：必须写。它防止利益相关方对交付范围产生错误预期。
