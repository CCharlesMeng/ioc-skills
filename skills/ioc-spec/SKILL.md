# 需求规格

## 概述

这个 skill 把视觉解码产物（Sketch-Decode.md）和需求文档交叉分析，产出两份面向产品和用户的文档：

| 产出 | 回答的问题 |
| --- | --- |
| `Spec.md` | 这个需求要做什么、不做什么、涉及谁 |
| `Acceptance.md` | 怎样算做完了（GWT 验收场景） |

这两份文档不包含任何技术实现细节，面向产品经理、测试人员和业务验收方。

内部有 5 个 Phase：

| Phase | 做什么 | 产出 |
| --- | --- | --- |
| 0 — 输入装载 | 读取并校验所有输入 | 输入完整性确认 |
| 1 — 需求地图 | 交叉对照 UI 与文档，识别缺口 | 需求地图表 |
| 2 — 需求澄清 | 覆盖度扫描 + 优先级提问 + 多轮收敛 | 澄清记录 + 更新后的需求地图 |
| 3 — 规格生成 | 产出 Spec.md | `Spec.md` |
| 4 — 验收生成 | 产出 Acceptance.md | `Acceptance.md` |

## 交互规范

| Phase | 用户可见内容 | 结束方式 |
| --- | --- | --- |
| 0 — 输入装载 | 内联进 Phase 1（P6） | — |
| 1 — 需求地图 | 需求地图表 + 缺口分类 | 确认门（P4） |
| 2 — 需求澄清 | 覆盖度扫描结论 + 逐条提问（含选项推荐） | 等待回答（多轮） |
| 3 — 规格生成 | Spec.md 内容预览 | 确认门（P4） |
| 4 — 验收生成 | Acceptance.md 内容预览 | 确认门（P4） |
| 最终 | 三行索引（P5） | — |

---

## 硬门禁

1. **没有 Sketch-Decode.md 不准分析。** 如果用户没有提供视觉解码产物（Sketch-Decode.md 或导出 DSL JSON），阻断并提示先运行 ioc-sketch-decode。如果确实没有设计稿（纯文字需求），在 Spec.md 中标注"本需求无设计稿输入"，所有 FR 仅基于需求文档。
2. **不要编造 UI 细节。** Sketch-Decode.md 中没有出现的界面元素，不允许在 Spec.md 或 Acceptance.md 中描述。需求文档提到但截图中未体现的功能，只记录文字需求，标记 `[待确认:无视觉依据]`。
3. **不要写技术术语。** Spec.md 和 Acceptance.md 中不允许出现 API、数据库、框架、接口、字段类型等技术概念。所有描述使用"用户看到 / 用户操作 / 系统响应"的语言。
4. **每条需求必须可溯源。** 功能需求的"来源"列必须指向 Sketch-Decode.md 的具体截图/组件，或需求文档的具体章节。无法溯源的条目标记 `[待确认:无视觉/文档依据]`。
5. **不要在澄清未收敛时生成规格。** Phase 2 的所有 🔴 必须变 🟢 才能进入 Phase 3。用户提前终止时，未澄清项必须在 Spec.md 中显式标记为 `unresolved` 或 `assumed`。

## 何时使用

适用场景：

- 已有 Sketch-Decode.md（ioc-sketch-decode 产出），需要分析并产出需求规格
- 已有导出 DSL JSON（设计平台直接导出），可直接作为结构化输入
- 已有需求文档（通过 clouddevops-mcp 可读取），需要与设计稿交叉验证
- 纯文字需求（无设计稿），直接基于需求文档产出规格

不适用场景：

- 需要基于现有代码进行技术分析和目标行为拆解（不在 ioc 流程范围内）
- 需要产出技术验收标准和测试规格（不在 ioc 流程范围内）
- 还没有设计稿的视觉解码（先用 `ioc-sketch-decode`）

## 前置条件

- `docs/<需求名>/Sketch-Decode.md` 已产出（如有设计稿），或有导出 DSL JSON 可供使用
- clouddevops-mcp 可访问（用于读取需求文档）

## 工作流

### Phase 0 — 输入装载

读取所有输入源，校验完整性。此阶段不产生独立输出，结论内联进 Phase 1（P6）。

动作：

1. **读取 Sketch-Decode.md**
   - 路径：`docs/<需求名>/Sketch-Decode.md`
   - 校验 4 节 schema 是否完整（布局还原 / 组件识别 / 交互与状态 / 解析自检）
   - 如果缺失 → 检查是否有导出 DSL JSON 可用（见下一步）
   - 如果某节缺失或为空 → 标记，在 Phase 1 中作为缺口处理

2. **读取导出 DSL JSON**（如有）
   - 设计平台/低代码平台导出的页面结构 JSON
   - 如果同时有 Sketch-Decode.md → 作为补充数据源，交叉校验布局结构
   - 如果没有 Sketch-Decode.md → 作为主数据源，从 JSON 中提取页面结构、组件和数据字段
   - 无论哪种情况，导出 JSON 中的 `components`、`columns`、`data` 等结构化信息均参与需求点提取

3. **读取需求文档**
   - 通过 clouddevops-mcp 读取相关需求文档
   - 提取结构化内容：功能描述、用户角色、业务规则、约束条件

4. **接收用户需求描述**
   - 用户在对话中提供的补充说明
   - 作为补充输入参与交叉引用

退出条件：至少有一个输入源（Sketch-Decode.md、导出 DSL JSON 或需求文档）可用。如果三者均无 → 阻断："请先提供设计稿（运行 ioc-sketch-decode）或需求文档"。

---

### Phase 1 — 需求地图

将所有输入源交叉对照，产出需求地图。

动作：

1. **提取需求点**
   - 从需求文档中提取每个功能描述
   - 从 Sketch-Decode.md 中提取每个可交互元素和数据展示
   - 从导出 DSL JSON 中提取组件结构和数据字段（如有）
   - 合并为统一的需求点列表

2. **交叉引用**
   - 每个需求点标注来源：文档来源（章节号）+ UI 来源（截图号 + 组件）
   - 识别三类缺口：

| 缺口类型 | 含义 | 处理方式 |
| --- | --- | --- |
| 有文档无 UI | 需求文档提到但设计稿未体现 | Phase 2 澄清：是未设计还是刻意省略 |
| 有 UI 无文档 | 设计稿有但需求文档未提及 | Phase 2 澄清：是隐含需求还是设计多余 |
| 两边都模糊 | 两边都提到但描述矛盾或不完整 | Phase 2 澄清：以谁为准 |

3. **产出需求地图表**

```markdown
| # | 需求点 | 文档来源 | UI 来源 | 状态 |
|---|--------|----------|---------|------|
| 1 | ... | 需求文档 X.X 节 | Sketch-Decode 截图N: 组件名 | 已对齐 / 缺UI / 缺文档 / 冲突 |
```

**确认门**：

```
---
**[Phase 1 确认门]** 以上是需求地图和识别的缺口。
→ 请确认继续 / 或指出需要修改的地方。
---
```

---

### Phase 2 — 需求澄清

Phase 1 的需求地图只解决了"文档和 UI 对不对得上"的问题。Phase 2 要更深一层：即使对上了，需求本身是否足够清晰、完整、可验收？

这是整个 skill 最关键的 Phase。它决定了 Spec.md 和 Acceptance.md 的质量上限。

#### 步骤 1：覆盖度扫描

对需求地图中的所有需求点，按以下 7 个维度逐一扫描，标记每个维度的覆盖状态：清晰 / 部分 / 缺失。

| 维度 | 检查什么 | 典型问题 |
| --- | --- | --- |
| 功能范围与边界 | 做什么、不做什么、边界在哪 | "支持批量操作"——批量上限是多少？哪些操作支持批量？ |
| 用户角色与权限 | 谁能做、谁不能做、角色差异 | 所有人都能新建工单吗？谁能关闭工单？ |
| 业务规则 | 约束条件、校验规则、限制 | 标题有字数限制吗？同一工单能重复提交吗？ |
| 交互流程 | 主流程 + 分支路径 + 中断恢复 | 提交后能撤回吗？填到一半刷新页面内容还在吗？ |
| 数据生命周期 | 创建、修改、归档、删除 | 已关闭的工单能重新打开吗？数据保留多久？ |
| 异常与边界 | 空态、大量数据、并发、网络异常 | 搜索无结果时显示什么？1000 条工单时分页怎么表现？ |
| 术语一致性 | 文档与 UI 用词是否统一 | 文档说"任务"，UI 写"工单"——以谁为准？ |

产出内部覆盖度表（不输出给用户，用于下一步生成问题）：

```
| 维度 | 状态 | 缺口说明 |
|------|------|----------|
| 功能范围与边界 | 部分 | "不做的事"未明确 |
| 用户角色与权限 | 缺失 | 未提及角色差异 |
| 业务规则 | 部分 | 必填校验有，其他校验规则未知 |
| ...  | ...  | ... |
```

#### 步骤 2：问题生成与优先级排序

综合两个来源生成候选问题：

1. **Phase 1 缺口**：需求地图中状态为"缺 UI / 缺文档 / 冲突"的条目
2. **覆盖度缺口**：步骤 1 中状态为"部分"或"缺失"的维度

对候选问题按 `影响面 × 不确定性` 排序，取 Top N（单轮最多 7 个，整个 session 最多 10 个）。

优先级规则：
- 影响功能范围边界的 > 影响单个交互细节的
- 角色/权限不明的 > 样式/文案不明的
- 两个输入源冲突的 > 单个输入源缺失的

每个问题标注三色状态：

| 状态 | 含义 |
| --- | --- |
| 🔴 | 缺失关键信息，无法合理推测，必须用户回答 |
| 🟡 | 有证据可推测，展示推理，用户可确认/纠正/跳过 |
| 🟢 | 已通过文档、UI 或上下文确认（不需要提问） |

#### 步骤 3：逐条提问

按优先级队列逐条向用户提问。每次只问一个问题，给出经过分析的选项，将推荐选项放在第一个并说明推荐原因。

提问格式：

```
**Q-1** 🔴 [维度: 用户角色] [来源: 缺文档依据]

哪些角色可以新建工单？

| 选项 | 说明 |
|------|------|
| **A. 所有登录用户均可新建** ⬅️ 推荐 | 设计稿未体现角色区分，新建按钮对所有用户可见；多数工单类系统采用此模式 |
| B. 仅特定角色可新建 | 适用于有明确分工的场景（如仅客服/运维可创建） |
| C. 其他 | 请描述你的方案 |

→ 请选择或补充。
```

提问规则：

- 🔴 问题优先于 🟡 问题
- 每个问题提供 2-4 个选项，最后一个始终为"其他"开放选项
- 推荐选项放在第一个，用 `⬅️ 推荐` 标记，说明栏给出推荐理由（基于设计稿证据、行业惯例或上下文推理）
- 🟡 问题的推荐选项即为 AI 的推测结论，用户可直接回车确认
- 用户回复后立即更新需求地图，再提下一个问题
- 整个 session 不超过 10 个问题

#### 步骤 4：逐条收敛

每条问题的处理循环：

1. 输出当前问题（含选项）
2. 用户回答
3. 更新需求地图（标注该需求点的澄清结论）
4. 更新覆盖度表
5. 检查是否还有下一个问题；如有，输出下一题

准出条件：
- 所有 🔴 已变为 🟢
- 所有 🟡 要么变为 🟢，要么被用户显式接受为工作假设（标记为 `🟡 已接受`，进入 Spec.md 时计入"约束与假设"）
- 覆盖度表中无"缺失"维度（"部分"可接受，但需在 Spec.md 中标注）

用户回答某个问题时，可能引发新的疑问——将新问题追加到队列尾部，但整个 session 不超过 10 个问题。

用户可随时说"够了"提前终止澄清。提前终止时，未澄清的 🔴 标记为 `unresolved`，🟡 标记为 `assumed`，均在 Spec.md 中显式声明风险。

#### 步骤 5：澄清结论记录

将所有澄清结论写入需求地图的附录，供 Phase 3/4 消费：

```
## 澄清记录

| Q# | 问题 | 结论 | 来源 |
|----|------|------|------|
| Q-1 | 哪些角色可以新建工单？ | 所有登录用户均可 | 用户确认 |
| Q-3 | 优先级是否只有三级？ | 是，高/中/低 | 🟡 已接受（工作假设） |
```

### Phase 3 — 规格生成

基于确认后的需求地图，生成 Spec.md。

结构模板见 [references/spec-template.md](references/spec-template.md)。

核心规则：

- **去技术化**：所有描述使用"用户看到 / 用户操作 / 系统响应"的语言
- **可溯源**：每条功能需求标注来源（Sketch-Decode 截图号 + 组件名，或需求文档章节号）
- **做与不做**：功能范围必须同时列出"做"和"不做"
- **待确认项**：无法溯源或存在假设的条目，使用 `[待确认:原因]` 标记

写入 `docs/<需求名>/Spec.md`。

**确认门**：

```
---
**[Phase 3 确认门]** 以上是需求规格说明。
→ 请确认继续 / 或指出需要修改的地方。
---
```

---

### Phase 4 — 验收生成

基于 Spec.md 的每条功能需求，生成 GWT 验收场景。

格式模板见 [references/acceptance-template.md](references/acceptance-template.md)。

核心规则：

- **FR 对应**：每个 AC 编号对应 Spec.md 的 FR 编号（AC-1 ↔ FR-1）
- **正向 + 异常**：每条功能需求至少一个正向场景 + 一个异常/边界场景
- **用户语言**：Given/When/Then 使用用户可理解的语言，不涉及技术实现
- **不编造 UI**：场景中引用的界面元素必须在 Sketch-Decode.md 中存在

交叉引用规则见 [references/cross-reference-rules.md](references/cross-reference-rules.md)。

写入 `docs/<需求名>/Acceptance.md`。

**确认门**：

```
---
**[Phase 4 确认门]** 以上是 GWT 验收场景。
→ 请确认继续 / 或指出覆盖缺口。
---
```

## 退出门禁

- 每条功能需求有来源标注
- 每条功能需求有至少一个 GWT 场景
- 无"实现时再看"的模糊表述
- 待确认项已显式标记，不会被误认为已确认的需求
- Spec.md 和 Acceptance.md 中无技术术语

## 降级模式

如果 Sketch-Decode.md 缺失（无设计稿）：

- 跳过交叉引用中的 UI 来源列
- Spec.md 中标注"本需求无设计稿输入，所有 FR 仅基于需求文档"
- Acceptance.md 的 GWT 场景不引用具体 UI 元素

如果需求文档缺失或不可读取：

- 跳过交叉引用中的文档来源列
- Spec.md 中标注"本需求无需求文档输入，所有 FR 仅基于设计稿解析"
- 建议用户补充文字需求描述

## 工件管理

| 工件 | 路径约定 |
| --- | --- |
| `Sketch-Decode.md` | `docs/<需求名>/Sketch-Decode.md`（前置输入，ioc-sketch-decode 产出） |
| `Spec.md` | `docs/<需求名>/Spec.md` |
| `Acceptance.md` | `docs/<需求名>/Acceptance.md` |

`<需求名>` 使用需求的短标识，中文或英文均可。

## 与下游的衔接

```
ioc-sketch-decode → ioc-spec → （由项目决定下一步）
```

ioc-spec 产出的 Spec.md + Acceptance.md 是面向产品和业务验收的完整文档（做什么、怎样算完成）。Phase 2 的需求澄清已覆盖功能范围、角色权限、业务规则、交互流程、数据生命周期、异常边界、术语一致性七个维度，下游无需重复澄清。

下一步由项目实际情况决定：

| 场景 | 下一步 |
| --- | --- |
| 有现有代码需要摸底（改造/迭代） | 进入技术分析，基于代码现状做目标行为拆解 |
| 全新项目无代码基础 | 直接进入技术设计或切片实现 |
| 仅需产品对齐 | Spec.md + Acceptance.md 即为最终产出 |

## 最终输出

```
✓ ioc-spec 完成
产出：docs/<需求名>/Spec.md, docs/<需求名>/Acceptance.md
下一步：<根据项目情况给出一条具体指令>
```
